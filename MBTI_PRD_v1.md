# 项目需求文档：AI 时代 MBTI 价值评估报告

## 1、文档目的与信息源
本文档用于将 `MBTI_background`、`MBTI_test_report` 与当前 PRD 合并为一份可直接用于产品、设计、研发、测试和内容协作的正式需求文档。

本版文档采用以下信息源分工：
- `MBTI_background`：作为业务背景、用户画像、引导文案、正式题库与测评定位的主要来源。
- `MBTI_test_report`：作为报告结构、语气、模块表达方式与内容颗粒度的样例参考。
- Figma 报告设计原型：作为页面层级、视觉布局、模块呈现顺序的设计参考。
- 本 PRD：作为最终工程实现约束；当上述材料存在冲突时，以本 PRD 的统一口径为准。

本次整合后，重点修正以下冲突：
- 背景稿中的“20分/维度”与“每题5分、每维度5题”存在口径冲突；工程实现统一按“命中次数判定”为准，不向用户展示分值。
- 背景稿、测试报告与 PRD 存在多个标题名称；本版统一定义命名体系，避免前后端和内容侧混用。
- 原 PRD 的“summary + items”结构与测试报告样例不一致；本版统一为“模块标题 + 3条判断项”的结果结构。

## 2、产品背景与目标人群

### 2.1 测评主题
- 测评主题：`16 型人格（MBTI）未来竞争力深度测评`
- 核心命题：在 AI 时代，通过 MBTI 类型判断用户的人性护城河、协作优势、潜在替代风险与跃迁路径。

### 2.2 目标人群
- 职场进阶者：担心技能被 AI 覆盖，急需挖掘自身人性护城河的专业人士。
- 自媒体创业者：寻找与天性契合的差异化定位，追求高辨识度个人 IP 的创作者。
- 终身学习者：希望根据性格底色优化知识内化效率，拒绝盲目跟风的学习者。

### 2.3 共性痛点
- 同质化焦虑：技能壁垒正在瓦解，不知道除了工具，自己还剩下什么核心价值。
- 变现困境：拥有性格标签却无法转化成商业收益，缺乏性格与市场的连接点。
- 内耗陷阱：在不擅长的领域死磕，误把天性中的特质当成成长的阻碍。

### 2.4 产品目标
1. 验证测评模块对“判断型报告”的承载能力，确保在统一框架下仍能输出差异化内容。
2. 帮助用户把 MBTI 从“性格标签”转化为“AI 时代的竞争力语言”。
3. 提升用户完成率、报告阅读深度与分享意愿。
4. 为后续课程、认证、咨询等承接能力预留空间，但不在本版流程中做显性营销。

## 5、测评结构与正式题库

### 5.1 测评维度
本测评基于四个 MBTI 核心维度进行判定：
- 维度一：能量来源（`E / I`）
- 维度二：信息过滤（`S / N`）
- 维度三：决策依据（`T / F`）
- 维度四：生活掌控（`J / P`）

每个维度 `5` 道题，共 `20` 道题。所有题目均为单选题，选项仅有 `A / B` 两个。

### 5.2 题目展示与提交规则
- 题目固定按 `Q01-Q20` 顺序展示，不做随机排序。
- 用户必须完成全部 `20` 道题后才允许提交。
- 每题仅接收 `A` 或 `B`，其他值视为非法。
- 前端不得根据题干推断字母映射，必须以后端下发配置为准。
- 题库与字母映射以本节为唯一正式版本。

## 6、判定规则与分值口径

### 6.1 最终判定方式
工程实现统一使用“字母命中次数”进行判定：
- 每题根据配置将 `A / B` 映射为一个维度字母。
- 每个维度共 `5` 题，统计两个字母的命中次数。
- 任一字母命中次数 `>= 3`，即判定为该维度最终结果。
- 四个维度按 `E/I -> S/N -> T/F -> J/P` 顺序拼接，生成最终 MBTI 类型。

### 6.2 分值口径统一说明
背景材料中存在两种分值说法：
- “每维度 20分”
- “每题 5分、每维度 5题”

二者在数学上不一致，因此本版统一处理如下：
- 用户侧不展示任何分数、比例、等级。
- 系统实现与测试验收均以“多数命中判定”作为正式逻辑。
- 若底层通用测评引擎必须保留分值字段，则仅允许内部兼容使用 `A=5, B=0`，对应维度满分 `25`，该口径不得对外展示。

### 6.3 结果边界
最终结果只允许为 `16` 种标准类型之一：
`ISTJ`、`ISFJ`、`INFJ`、`INTJ`、`ISTP`、`ISFP`、`INFP`、`INTP`、`ESTP`、`ESFP`、`ENFP`、`ENTP`、`ESTJ`、`ESFJ`、`ENFJ`、`ENTJ`

## 7、报告生成规则与样例对齐

### 7.1 生成方式
报告采用 `LLM 动态生成`，在用户提交答卷并得到 `mbtiType` 后一次性生成完整报告。

生成输入仅允许包含：
- `mbtiType`
- 测评正式标题
- 报告结构定义
- 模块写作约束
- 禁用词和禁用表达
- 经授权的认证方向白名单

生成输入不得包含：
- 用户逐题答案明细
- 维度得分或命中次数
- 任何百分比、排名、等级
- 课程销售文案

### 7.2 报告结构
以 `MBTI_test_report` 为结构和语气参考，报告页固定包含四个模块，顺序不可变：
1. 性格资产总评
2. 你的天才所在
3. 替代风险
4. 职业认证与阶层跃迁建议

每个模块统一为：
- 1 个模块标题
- 3 条判断项

本版不再使用 `summary + items[]` 的双层结构，统一简化为“模块标题 + 3条独立判断”。

### 7.3 模块内容要求

| 模块key | 模块标题 | 条目数 | 内容要求 |
| --- | --- | --- | --- |
| asset_overview | 性格资产总评 | 3 | 第1条必须明确用户 MBTI 类型及 AI 协作角色；第2条写人性护城河；第3条写组织/系统中的协作位置。 |
| talent_strengths | 你的天才所在 | 3 | 3条都必须是“可被现实场景感知、可被 AI 放大、但不依赖 AI”的能力判断。 |
| substitution_risks | 替代风险 | 3 | 前2条写结构性风险；第3条必须给出明确 AI 协作建议或提效建议。 |
| career_path | 职业认证与阶层跃迁建议 | 3 | 第1条写适配认证/能力方向；第2条写角色升级方向；第3条写可执行跃迁路径。 |

### 7.4 字数与颗粒度
为与 `MBTI_test_report` 的表达颗粒度和截图传播场景保持一致，统一采用较短报告形态：

| 模块 | 字数范围 |
| --- | --- |
| 性格资产总评 | 110-160字 |
| 你的天才所在 | 110-160字 |
| 替代风险 | 90-140字 |
| 职业认证与阶层跃迁建议 | 110-170字 |

字数统计口径：
- 仅统计模块正文，不统计模块标题。
- 统计中文、英文、数字、标点。
- 每条判断项建议为单句或双分句，不写成长段。

### 7.5 样例报告使用规则
`MBTI_test_report` 为 `ESFJ` 示例，仅用于以下目的：
- 校验模块顺序是否正确。
- 校验每模块是否为 `3` 条内容。
- 校验内容语气是否为判断型、资产型、路径型表达。
- 校验“风险模块包含 AI 建议”“跃迁模块包含认证方向”的写法。

样例报告不作为以下内容的固定模板：
- 不要求逐字复用。
- 不要求所有类型都沿用相同句式。
- 不要求所有类型都绑定 `CAIE`，但允许在白名单存在时使用 `CAIE` 作为认证方向示例。

### 7.6 内容红线
报告禁止出现以下内容：
- MBTI 理论解释、人格科普、测试免责声明。
- 模型自述、AI 自述、提示词说明、生成过程说明。
- “可能”“也许”“大概”“因人而异”“视情况而定”等概率化表达。
- 分数、比例、等级、打败百分之多少用户等描述。
- 直接复制 `MBTI_test_report` 的完整句子。
- 与本测评无关的营销导流、价格信息、联系方式。

### 7.7 生成校验与失败重试
任一情况发生即视为生成失败：
- 非法 JSON。
- 缺少 `mbtiType`、`reportTitle`、`modules` 任一顶层字段。
- 模块数量不等于 `4`。
- 模块顺序错误或标题错误。
- 任一模块条目数不等于 `3`。
- 任一模块字数不在规定区间。
- 命中禁用表达。

失败策略：
- 单次报告生成最多重试 `2` 次。
- 若仍失败，则返回统一异常态。
- 失败结果不得缓存，不得进入历史记录。

## 10、缓存、历史记录与异常规则

### 10.1 缓存规则
- 用户首次提交成功后，先完成 MBTI 判定，再触发报告生成。
- 报告生成成功后，缓存完整报告正文。
- 再次进入结果页时优先读取缓存，不重复调用 LLM。

### 10.2 历史记录规则
历史记录至少展示：
- 测评名称
- 生成时间
- `mbtiType`

本版不展示分值、维度比例、完整报告摘要。

### 10.3 异常规则
- 漏答、非法题号、非法选项值：提交失败，不生成结果。
- MBTI 判定成功但报告生成失败：展示统一异常态，不展示部分内容。
- 异常结果不缓存、不入历史。
- 同一 `recordId` 若已有成功结果，则返回缓存版本，不重新生成第二份报告。

## 11、交付范围与验收标准
- [ ] 入口页文案、开始语、结束语、分享文案与背景材料一致。
- [ ] 题库与 `MBTI_background` 中的 `20` 道正式题完全一致。
- [ ] 系统可基于答题结果稳定生成唯一 MBTI 类型。
- [ ] 报告标题、传播主题、正式标题的命名口径与本 PRD 一致，不混用。
- [ ] 报告结构与 `MBTI_test_report` 一致：四个固定模块、每模块 `3` 条判断项。
- [ ] 报告内容不出现分数、比例、MBTI 理论说明、AI 自述、概率化措辞。
- [ ] 风险模块第 `3` 条必须是 AI 协作建议；跃迁模块必须包含认证或能力方向建议。
- [ ] 页面展示顺序与 Figma 原型一致。
- [ ] 报告成功后缓存命中；失败结果不缓存、不进历史记录。

## 12、测试用例清单

### 12.1 判题测试
- 全量 `A` 答卷应输出 `ESTJ`。
- 全量 `B` 答卷应输出 `INFP`。
- 构造各维度 `3:2`、`4:1` 边界答卷，验证判定正确。
- 提交漏答、重复题目、非法题号、非法选项值时，应被服务端拦截。

### 12.2 报告结构测试
- 报告返回必须包含 `themeTitle`、`reportTitle`、`mbtiType`、`modules`。
- 模块数固定为 `4`，每模块条目数固定为 `3`。
- 模块顺序必须为：总评 -> 天才 -> 风险 -> 跃迁。
- 各模块字数符合本 PRD 约束。

### 12.3 样例一致性测试
- 使用 `ESFJ` 样例生成报告时，应在结构和语气上接近 `MBTI_test_report`，但不得逐字复制。
- 样例中体现的“角色定位 / 能力判断 / 风险与 AI 建议 / 认证与路径”四类信息，其他类型也必须完整具备。

### 12.4 页面与异常测试
- 页面呈现顺序符合原型。
- 报告成功后再次进入结果页，应命中缓存。
- 模拟 LLM 超时、字段缺失、字数超限、条目不足时，应触发重试与最终异常态。

## 13、本版默认假设
- 既有测评模块主体已存在，本 PRD 仅补充本测评的业务与工程约束。
